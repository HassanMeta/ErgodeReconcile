2. GETTING STARTED
================================================================================

SYSTEM REQUIREMENTS:

- Modern web browser (Chrome, Firefox, Edge, Safari)
- Access to the Ergode Reconciliation System
- CSV or Excel files with your transaction data

BEFORE YOU BEGIN:

You'll need:
1. Credit card statement files (CSV or Excel format)
   - Transaction Date (MM/DD/YYYY format)
   - Credit Card Number
   - Transaction Description
   - Transaction Amount
   - Unique Reference ID for each transaction

2. Purchase order files (CSV or Excel format)
   - PO Date
   - PO Number
   - Vendor Code/Prefix
   - PO Amount
   - Department (FBA or FBM)

3. Vendor master data (should be set up by administrator)
   - Vendor Prefix/Code
   - Vendor Name
   - Category (ONLY FBA, ONLY FBM, COMMON, NOT AVBL)
   - Department
   - Payment Terms
   - CC Fee Rate

FIRST TIME SETUP:

1. Open the application in your web browser
2. The system will start on the "Masters" page
3. Verify that vendor master data is set up (contact administrator if needed)
4. You're ready to begin!

This tab manages vendor master data - the core information about each vendor.

VIEWING EXISTING VENDORS:

1. The "Existing Vendor Master Entries" table shows all vendors
2. You can see:
   - PREFIX: Vendor code/identifier
   - VENDOR_NAME: Full vendor name
   - CATEGORY: Vendor category (ONLY FBA, ONLY FBM, COMMON, NOT AVBL)
   - DEPT: Department (FBA or FBM)
   - PAYMENT TERMS: Payment terms in days (0, 10, 15, or 30)
   - CC FEE: Credit card fee rate (e.g., 0.03 for 3%)

EDITING EXISTING VENDORS:

1. Find the vendor in the table
2. Click directly in the "PAYMENT TERMS" or "CC FEE" column
3. Make your changes
4. Changes are saved automatically when you click outside the cell

Note: PREFIX, VENDOR_NAME, CATEGORY, and DEPT cannot be edited inline.
Use the "Add Vendor Master Entry (Legacy)" form to add new vendors.

ADDING A NEW VENDOR (Legacy Form):

1. Click the "Add Vendor Master Entry (Legacy)" expander
2. Fill in the form:
   - Vendor Prefix: Enter vendor code (max 20 characters, stored in uppercase)
   - Vendor Name: Enter full vendor name (max 100 characters)
   - Category: Select from dropdown:
     * ONLY FBA: Vendor only for FBA department
     * ONLY FBM: Vendor only for FBM department
     * COMMON: Vendor used by both departments (you'll assign dept later)
     * NOT AVBL: Vendor not available for reconciliation
   - Dept: Select FBA or FBM
   - Payment Terms (Days): Select payment terms:
     * 0 = Pre-payment (0 days)
     * 10 = Net 10
     * 15 = Net 15
     * 30 = Net 30
   - CC Fee Rate: Enter fee rate (e.g., 0.03 for 3%, 0.025 for 2.5%)
3. Click "Save Vendor" button
4. You'll see a success message if saved correctly

IMPORTANT NOTES:

- Vendor Prefix must be unique
- If you try to add a duplicate prefix, you'll get an error
- All prefixes are stored in uppercase automatically
- CC Fee Rate is a decimal (0.03 = 3%, not 3)

   - DESCRIPTION: The transaction description text
   - PREFIX: The vendor prefix this description maps to

ADDING A NEW MAPPING:

IMPORTANT:

- Extra spaces are automatically normalized
- If a description already exists, you'll get an error
- The system matches descriptions exactly (after normalization)

STEP 1: SELECT A BATCH

1. At the top of the page, you'll see a dropdown: "Select CC Batch to Reconcile"
2. The dropdown shows all available CC batches with row counts
   Example: "CCBATCH-1234-12252024 (25 rows)"
3. Select the batch you want to reconcile

STEP 2: AUTOMATIC RECONCILIATION

Once you select a batch:
- The system automatically runs the reconciliation
- You'll see a spinner: "Running reconciliation for batch..."
- The reconciliation process:
  * Matches CC transactions to vendors using description mappings
  * Finds matching purchase orders based on date windows
  * Calculates totals, flags, and deductions
  * Groups results by date, vendor, department, and payment terms

STEP 3: VIEW RESULTS

After reconciliation completes, you'll see multiple tabs with different views.

APPLYING PO DEDUCTIONS:

PO deductions allow you to apply amounts from purchase orders to credit card

1. In the PO Details table, find the PO you want to deduct from
2. In the "Deduct" column, enter the amount you want to apply

4. Enter an optional reason in the "Reason" field (placeholder text)

VALIDATION RULES:

- Amount cannot be zero

- Negative amount cannot exceed the total amount already applied
- Example: If $100 was applied, you can correct up to -$100, but not -$150

UNDERSTANDING PO DEDUCTIONS
â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”

PO deductions are amounts applied from purchase orders to credit card batches.
They persist across all future reconciliations.

KEY CONCEPTS:

Original_PO_Amount:
- The original amount of the PO (never changes)
- This is what was originally entered

Applied_Amount (Total_Deductions):
- Sum of all deductions applied to this PO
- Can be positive (additions) or negative (corrections)
- Cumulative across all reconciliations

Available Balance:
- Original_PO_Amount - Applied_Amount
- The amount still available for future deductions

Adjusted_PO_Amount:
- Used for reconciliation calculations
- = Applied_Amount if > 0, else Original_PO_Amount

EXAMPLE:

PO Number: PO-12345
Original Amount: $1,000.00

Reconciliation 1:
- Apply $300.00
- Applied_Amount: $300.00
- Available: $700.00

Reconciliation 2:
- Apply $200.00
- Applied_Amount: $500.00 (cumulative)
- Available: $500.00

Reconciliation 3:
- Made a mistake, need to correct
- Apply -$100.00 (negative)
- Applied_Amount: $400.00 (corrected)
- Available: $600.00

CORRECTING MISTAKES:

If you applied the wrong amount:

1. Go to Analysis tab for the vendor
2. Find the PO in PO Details table
3. Enter a negative amount in the "Deduct" column
   - Example: If you applied $900 but meant $90, enter -$810
4. Enter a reason (optional)
5. Click "Apply All Deductions"
6. The system will validate and apply the correction

VALIDATION FOR CORRECTIONS:

- You cannot apply a negative amount if nothing has been applied yet
- Negative amount cannot exceed the total amount already applied
- Example: If $100 applied, you can correct up to -$100, but not -$150

COMMON ISSUES AND SOLUTIONS
â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”

Q: My file upload failed. What should I do?

A: Check the following:
   - File format: Must be CSV or Excel (.csv, .xlsx, .xls)
   - Date format: Must be MM/DD/YYYY (e.g., 12/25/2024)
   - Required columns: Make sure all required fields are present
   - Column mappings: Verify you mapped all required columns correctly
   - Data types: Amounts must be numbers, dates must be valid dates

Q: Transactions are showing as "Unmapped". Why?

A: This means the transaction description doesn't match any mapping:
   - Go to Masters â†’ Description Mapping
   - Check if a similar description exists (case doesn't matter)
   - Add a new mapping if the vendor exists
   - If vendor doesn't exist, contact administrator to add vendor

Q: I'm getting "Red Flags" everywhere. What's wrong?

A: Red Flags mean CC amount > PO amount. Common causes:
   - Missing purchase orders (POs not uploaded yet)
   - PO deductions need to be applied
   - Date window too narrow (increase Grace Days)
   - Vendor codes don't match between PO and Master data
   - Check Analysis tab to see which POs matched

Q: How do I fix a wrong PO deduction?

A: Use negative amounts to correct:
   1. Go to Reco â†’ Analysis tab
   2. Select the vendor
   3. Find the PO in PO Details
   4. Enter negative amount (e.g., -100 to reduce by $100)
   5. Enter reason
   6. Click "Apply All Deductions"
   Note: You can only correct up to the amount already applied

Q: Why can't I apply a negative deduction?

A: Negative deductions have restrictions:
   - You must have already applied some amount first
   - Negative amount cannot exceed total amount already applied
   - Example: If $50 applied, you can correct up to -$50, not -$100

Q: My vendor code doesn't match. What should I do?

A: Vendor codes must match exactly:
   - Check Masters â†’ Vendor Master for correct prefix
   - Vendor codes are case-insensitive but must match otherwise
   - Update your PO file with correct vendor codes
   - Or add the vendor to Masters if it doesn't exist

Q: The reconciliation is taking too long. Is this normal?

A: Processing time depends on:
   - Number of transactions in the batch
   - Number of purchase orders
   - Complexity of matching logic
   - Large batches (1000+ transactions) may take 30-60 seconds
   - This is normal - wait for the spinner to complete

Q: I uploaded the wrong file. How do I remove it?

A: Use the rollback feature:
   1. Go to PO Data or CC Data page
   2. Click "Manage Imports" expander
   3. Select the batch you want to remove
   4. Click "Rollback Selected Batches"
   WARNING: This cannot be undone!

Q: Can I edit vendor information after it's created?

A: Yes, but limited:
   - You can edit Payment Terms and CC Fee inline in the table
   - PREFIX, VENDOR_NAME, CATEGORY, and DEPT cannot be edited inline
   - To change those, you may need to contact administrator or use legacy form

Q: What does "Grace Days" mean?

A: Grace Days is a buffer period for matching POs to CC transactions:
   - Default: 5 days
   - For payment terms > 0: Looks back (CC Date - Payment Terms - Grace Days)
   - For payment terms = 0: Looks (CC Date - Grace Days) to (CC Date + Grace Days)
   - Increase if POs are dated further from CC transactions
   - Decrease for stricter matching

Q: How are PO deductions cumulative?

A: PO deductions persist across all reconciliations:
   - When you apply $100 to PO-123, it's saved permanently
   - Next reconciliation: Available = Original - $100
   - If you apply another $50: Available = Original - $150
   - Deductions are tracked by CC Batch ID for audit trail

Q: What's the difference between "Download Reco" and sidebar download?

A:
   - Sidebar Download: Single batch, all fields, quick access
   - Download Reco Page: Multiple batches, selected columns only, optimized for Excel

Q: Why are some transactions automatically processed in Common tab?

A: Transactions under $100 are automatically assigned to FBM department:
   - This saves time on small transactions
   - You only need to review transactions $100 and above
   - This is a system default to improve efficiency

Q: Can I change the automatic $100 threshold?

A: No, this is a system default. Contact administrator if you need this changed.

Q: What happens if I don't process Common transactions?

A: They remain in "Common" status:
   - They won't appear in Results summary
   - They won't be included in reconciliation calculations
   - You should process them to complete the reconciliation

Q: How do I know which POs matched which CC transactions?

A: Use the Analysis tab:
   1. Select a vendor
   2. Go to "PO Details" subtab to see matched POs
   3. Go to "CC Transaction Details" subtab to see CC transactions
   4. The date windows show the matching logic

Q: The download file has too many columns. Can I get fewer?

A: The "Download Reco" page already provides optimized columns:
   - Only essential fields for analysis
   - PO_Number and CC_Reference_ID as comma-separated lists
   - Perfect for Excel filtering and VLOOKUP

"Insufficient balance. Available: $X, Requested: $Y"
- You're trying to apply more than what's available
- Check the Available column in PO Details
- Reduce the deduction amount

"Cannot apply negative amount. No amount has been applied yet."
- You're trying to correct a deduction that doesn't exist
- Check deduction history to see what's been applied
- You can only correct existing deductions

"Cannot reduce applied amount by $X. Only $Y has been applied."
- Your negative amount exceeds what was applied
- Maximum correction = total amount already applied
- Example: If $100 applied, max correction is -$100

"PO {number} is fully applied. No more amount available."
- The PO has no remaining balance
- Check the Available column (should be 0 or negative)
- You cannot apply more to this PO

"Duplicate CC_Reference_ID values found"
- Your CC file has duplicate reference IDs
- Each transaction must have a unique CC_Reference_ID
- Remove duplicates from your file before uploading

"Vendor master file not found"
- Vendor master data hasn't been set up
- Contact administrator to set up vendor master data
- Reconciliation cannot work without vendor masters

"Unable to read CC data" or "Unable to read PO data"
- Data files may be corrupted
- Contact administrator to check system files
- May need to re-upload data

11. GLOSSARY OF TERMS
================================================================================

ADJUSTED_PO_AMOUNT:
The PO amount used for reconciliation calculations. Equals Applied_Amount if
deductions exist, otherwise equals Original_PO_Amount.

APPLIED_AMOUNT:
The cumulative total of all deductions (positive and negative) applied to a PO.
Also called Total_Deductions.

AVAILABLE BALANCE:
The remaining amount on a PO after deductions: Original_PO_Amount - Applied_Amount.

BATCH ID:
A unique identifier for a group of imported records. Format varies:
- PO batches: POBATCH-{timestamp}
- CC batches: CCBATCH-{last4digits}-{MMDDYYYY}

CC FEE:
Credit card fee rate, expressed as a decimal (e.g., 0.03 = 3%). Applied to
original PO amounts to calculate Total_CC_Charge.

CC_REFERENCE_ID:
A unique identifier for each credit card transaction. Must be unique across
all transactions.

CATEGORY:
Vendor classification:
- ONLY FBA: Vendor only for FBA department
- ONLY FBM: Vendor only for FBM department
- COMMON: Vendor used by both (requires department assignment)
- NOT AVBL: Vendor excluded from reconciliation

COMMON TRANSACTION:
A transaction that matched a vendor marked as "COMMON" and needs department
assignment (FBA or FBM).

COMPARISON_AMOUNT:
The amount used to compare CC transactions with POs. Equals Applied_Amount if
deductions exist, otherwise Original_PO_Amount.

DEDUCTION:
An amount applied from a PO to a CC batch. Can be positive (apply) or negative
(correct). Deductions are cumulative and persistent.

DEPT:
Department classification: FBA (Fulfillment by Amazon) or FBM (Fulfillment by
Merchant).

GRACE DAYS:
A buffer period (default 5 days) that extends the date window for matching POs
to CC transactions.

GREEN FLAG:
Indicates CC Amount â‰¤ PO Amount. Everything looks good, no action needed.

MAPPED TRANSACTION:
A CC transaction that was successfully matched to a vendor and has a clear
department assignment.

ORIGINAL_PO_AMOUNT:
The original amount of the purchase order. Never changes, even after deductions.

PAYMENT TERMS:
The number of days until payment is due:
- 0 = Pre-payment (payment before delivery)
- 10 = Net 10 (payment due in 10 days)
- 15 = Net 15 (payment due in 15 days)
- 30 = Net 30 (payment due in 30 days)

PREFIX:
Vendor code/identifier. Must match exactly between PO data and master data.

RED FLAG:
Indicates CC Amount > PO Amount. Needs investigation - may indicate missing PO,
amount mismatch, or need for deductions.

ROLLBACK:
Removing imported batches from the system. Cannot be undone.

TOTAL_CC_CHARGE:
The credit card fee amount calculated as: Original_PO_Amount Ã— CC_Fee.

TOTAL_DEDUCTIONS:
Same as Applied_Amount - the cumulative total of all deductions applied to a PO.

UNMAPPED TRANSACTION:
A CC transaction that could not be matched to any vendor. Usually means the
description doesn't exist in the mapping system.

VENDOR_PREFIX:
Same as PREFIX - the vendor code/identifier.

WINDOW (Date Window):
The date range used to match POs to CC transactions. Calculated based on:
- CC transaction date
- Payment terms
- Grace days
